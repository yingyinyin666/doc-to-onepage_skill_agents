# 旅行管家服务产品需求文档

## 基本信息
- 版本：1.0
- 摘要：创建
- 修改人：张雪立
- 时间：2026.03.16

## 一句话描述
在消费者购买线路游产品后为用户分配平台旅行管家，旅行管家拉上用户和商家服务人员组建企业微信，全程在群内提供服务。

## 1. 背景

### 1.1 旅行管家是什么
当用户购买线路游商品并完成预约后，将由平台建立行程群，群内角色包含用户、平台旅行管家、商家服务人员。行程中主要服务仍由商家服务人员提供，平台管家将在必要时对商家进行督促或配合商家共同解决用户问题。直至行程结束且用户无其他问题，将由平台管家统一解散群。

### 1.2 为什么要做旅行管家
用户在线上购买旅行产品时最担心的问题是线下履约的不确定性，旅行管家服务模式是降低用户消费顾虑、提升行程体验的重要手段。

#### 1. 从市场竞争上
旅游赛道已迈入价值深耕阶段，OTA巨头凭借深厚的供应链积淀与成熟服务体系稳居市场主导。抖音虽依托内容种草实现快速增长，但以团购、预售为主的模式存在服务深度短板；行业内普遍将旅行用户作为核心利润贡献群体，携程、飞猪、同程在旅行管家业务布局超5年并实现成熟运营与初步盈利，抖音在该服务上已处于落后态势（2024年GMV市场份额：携程55%、同程15%、美团13%、飞猪8%、抖音3%）。

#### 2. 从用户体验上
2026春节期间开始小范围试点，回收结论表现符合预期。用户对服务模式认可度高，管家服务满意度80.6%，用户模式认可度83%。

#### 3. 从商家经营上
管家服务可以提升商家经营效率、降低担责风险。流程管家线上服务，可大幅减少信息不对称引发的用户咨询与行程争议。

### 1.3 当前遇到了哪些问题

#### 问题1：服务建联成功率仅67%
- 客服号码外呼成功率约60%，与生服大盘外呼接通率基本一致。通过先短信再外呼的方式提升空间仍然不大，峰值时仅能达到70%左右。
- 用户添加链路复杂流失率高，企业微信发起添加申请后，需要在服务号红点中找到再按步骤添加，共有5个节点，用户整体完成率仅有80%左右。
- 用户对管家存在质疑、利益点不明确，用户从购买到预约过程无任何平台提示，预约成功外呼用户会产生不信任感甚至排斥。

#### 问题2：管家缺少端内服务平台
- 当前管家登陆为使用自有手机扫码，下班后可携带账号回家。服务时完全使用非抖音端内的企业微信电脑端，存在隐私风险。
- 服务结束后手动截图将聊天记录截图搬运回工单系统，管家操作效率较低对产能限制大。

#### 问题3：缺少科学的收益回收工具
- 从实验上缺少合理的分流动作，用户分流上缺少一致性。
- 从数据回收上主要通过人工统计+报表取数方式费力程度较高，时效性也无法保障。

### 1.4 如何解决问题

#### 解决方案1：系统自动通知、拉群，分发进群链接
订单支付后通知权益 --> 预约成功后系统自动创建订单微信群 --> 发送进群链接给平台管家、商家poc和客户 --> 群组内商家poc拉行程导游/地接/司机等 --> 开始正式服务

#### 解决方案2：客服工作台从端外切回端内
使用客服体系内天启系统，实现留证、聊天、质检、管理回到端内

#### 解决方案3：通过libra进行精准实验分流
使用ABtest工具进行用户切分，后续命中用户均通过该方式决策是否分流。

## 2. 目标

### 定性目标
1. 通过科学的线上化方式，协助项目完成业务收益回收
2. 实现旅行管家自动建联拉群能力，提高用户服务成功率
3. 实现旅行管家工作回归端内，提升管家工作效率和组织管理效率
4. 合规、隐私问题0发生

### 定量目标
1. 业务指标：核销率diff+6%、复购率diff+2.5%
2. 效率指标：服务成功率67%->80%、管家人月产能+12%

## 3. 需求

### 3.1 需求清单

| 模块 | 描述 | 优先级 |
|------|------|--------|
| 自动建群&通知 | 用户预约成功后自动建企业微信群 | P0 |
| 实验&服务通知 | 通过主动服务策略平台配置和Libra实验能力进行科学分流，并在关键节点对用户和管家进行通知 | P0 |
| 旅行管家详情页 | 展示管家服务模式和利益点，并将二维码嵌入在页面中 | P0 |
| 客服端内企微工作台 | 管家工作台切回使用抖音内「天启」系统 | P1 |

### 3.2 需求详情

#### 业务节点：订单支付成功
通过主动服务策略平台进行策略配置和实验分流：
- 策略触发规则：信号=支付成功、商品三级类目包含指定类目、用户标签=高传播or会员等级V7/V8or订单权益=安心游、行程是否包含购物=否、商家名单和商品名单需引用指定名单
- 策略触发限制：频控1天1次、服务过滤30天内无工单历史无赔付
- 服务标签：将实验组用户订单标记为【旅行管家权益订单】

通知用户预约成功后提供管家服务：
- 短信通知
- 站内信通知
- 团购小助手通知

#### 业务节点：预约成功
自动拉群：
- 群内成员：根据商家x管家匹配规则拉群
- 群名称默认昵称：抖音团购 x月xx日 服务群+订单号

通知用户和管家进群：
- 短信、站内信、团购小助手通知
- 飞书群通知（3种：用户进群前、进群后、超12小时未进群）

#### 旅行管家详情页
根据订单状态展示不同内容：
- 状态一：支付成功待预约（含状态回退）- 提示【预约成功后可进群】
- 状态二：预约成功 - 提示【已开启服务，请尽快进群】
- 状态三：出行后1天 - 提示【行程已开始，无法进群】
- 状态四：订单已退款、已核销 - 提示【订单状态变更，无法进群】

#### 企微工作台
使用抖音内「天启」系统，实现端内服务

## 4. 数据

### 4.1 实验要求
- 全流程使用同一个实验，基于uid分流
- 必须指标组：
  1. 预约：发起预约率、预约成功率、预约失败率
  2. 核销：订单x日核销率、新订单x日核销率
  3. 复购：x日复购率、同品类x日复购率
  4. 体验：广义&狭义CPO、指定三级类目CCR
